History & Purpose of Version Control

Before we touch a single Git command, let's understand the problem version control was built to solve — and how that solution evolved over 50 years.

The Problem

Life Before Version Control

📁

The Familiar Folder Mess

  • Syllabus_2023.docx
  • Syllabus_2023_revised.docx
  • Syllabus_2023_revised_FINAL.docx
  • Syllabus_2023_revised_FINAL_HOD_approved.docx
  • Syllabus_2023_revised_FINAL_HOD_approved(1).docx
💡

What We Actually Needed

  • One file, many tracked changes over time
  • Know who changed what, and why
  • Safely try a new idea without losing the original
  • Merge two people's edits automatically
  • Roll back to any earlier point, instantly
  • One single, unambiguous "current" version, always
⏱️

Cost: Time — hours lost manually comparing two documents line-by-line.

😰

Cost: Risk — one careless overwrite can destroy hours of someone's work, silently.

🤷

Cost: Trust — no way to prove who approved a change, or when.

Real world: Every department has lived this exact pain with exam paper drafts, accreditation (NAAC/NBA) documents, and shared research drafts. Version control is the engineering answer to this very human problem.

Pattern to notice: Each generation solved the previous generation's biggest pain point — single-file → whole-project, server-dependent → distributed, command-line-only → social & visual.
Categories

Three Types of Version Control Systems

1. Local VCS

A simple database on your own machine tracking file changes over time (e.g. RCS, 1982).

  • ✅ Simple, no setup needed
  • 🚫 No sharing, no backup off-machine
Like keeping numbered photocopies of your draft in a personal drawer — works alone, not for teams.

2. Centralized VCS

One central server holds the full history; everyone checks out from it (CVS, SVN).

  • ✅ One source of truth, simpler mental model
  • 🚫 Server down ⇒ nobody can commit or see history
Like a single master register kept in the HOD's office — if the office is locked (server down), nobody can work.

3. Distributed VCS

Every collaborator has a full copy of the history on their own machine (Git, Mercurial).

  • ✅ Works offline, no single point of failure
  • ✅ Any peer can become the new "server" instantly
Like every faculty member having a complete, identical copy of the register — work offline, sync later, no single point of failure.
Why It Exists

The Purpose of Version Control

🕒

History

Every change is timestamped, attributed and recoverable forever.

No more guessing which file is the "real" latest one.

🤝

Collaboration

Multiple people work on the same project without overwriting each other.

Git automatically merges non-conflicting edits to the same file.

🧪

Safe Experimentation

Try a risky change in isolation (a branch) without breaking the working version.

Delete the branch with zero consequence if the idea doesn't work out.

🛡️

Accountability & Backup

Know exactly who changed what and why — and never truly lose work.

Every clone is itself a full, independent backup of the whole history.

Real world: Three faculty co-authoring a research paper can each work in their own branch, see exactly which paragraph a co-author changed, and merge everything into one final draft — with the ability to undo any single change.

The Story of Git

Why did the world need yet another version control system in 2005, when CVS and SVN already existed?

Origin Story

Born From a Crisis — April 2005

The Linux Kernel Problem

The Linux kernel — thousands of contributors worldwide — used a proprietary tool called BitKeeper to manage its source code for free.

In April 2005, the company withdrew the free license after a dispute. Linus Torvalds and the kernel team suddenly had no version control system for the world's largest collaborative codebase.

Linus's response: build a new one from scratch. In about 10 days, the first working version of Git existed.

🧑‍💻

Design Goals Linus Set

  • Must be fast (kernel has 1000s of files)
  • Fully distributed — no single server to lose
  • Strong integrity guarantees against corruption/tampering
  • Cheap, instant branching
  • Support fully non-linear workflows (thousands of parallel contributors)
Fun fact: "Git" is British slang for an unpleasant person. Linus joked: "I'm an egotistical bastard, and I name all my projects after myself. First Linux, now Git."
The Differentiator

Why Git, When Others Already Existed?

⚡ Speed

Most operations (commit, diff, log, branch) happen locally — no network round-trip.

The Linux kernel has 70,000+ files — Git still handles a commit in milliseconds.

🌐 Truly Distributed

Every clone is a complete backup with full history — no "the server is down" moment.

Commit, branch and review history on a flight with no Wi-Fi; sync once you land.

🌿 Cheap Branching

In CVS/SVN, branching was slow and "scary." In Git it takes milliseconds — so people actually use it.

A branch is just a 41-byte pointer to a commit — not a copy of the whole project.

🔒 Data Integrity

Every snapshot is checksummed (SHA hash) — silent corruption is virtually impossible.

Change one character anywhere in history, and every hash after it changes too.

Real world analogy: Centralized tools were like a single shared whiteboard photographed by a clerk. Git gave everyone their own whiteboard that perfectly syncs with everyone else's — work anywhere, even with no Wi-Fi.

Step 1

Setting Up Git — Tell It Who You Are

git config --global user.name "Harinath K"
Sets the name attached to every commit you make on this machine.
git config --global user.email "you@vistas.ac.in"
Sets the email attached to every commit — should match your GitHub account email.
git config --list
View all current configuration settings.
git config --global init.defaultBranch main
Sets "main" as the default name for new repositories' first branch.
$ git config --global user.name "Harinath K" $ git config --global user.email "harinath@vistas.ac.in" $ git config --list user.name=Harinath K user.email=harinath@vistas.ac.in init.defaultbranch=main ← this machine is ready to commit
Real world: This is exactly like writing your name and signature format once, before signing every exam answer script you evaluate — every commit is "signed" with this identity, forever.
Starting Out

git init vs. git clone

🆕 git init

git init
Turns the current folder into a brand-new, empty Git repository.

Use this when starting a fresh project from nothing.

$ mkdir nba-accreditation $ cd nba-accreditation $ git init Initialized empty Git repository in .../nba-accreditation/.git/
Like opening a brand-new register and writing "Page 1" — starting your own course notebook from scratch.

📥 git clone

git clone <repo-url>
Downloads a complete copy of an existing repository — files and full history.

Use this to start from a project that already exists on GitHub.

$ git clone https://github.com/vistas/lab-manual.git Cloning into 'lab-manual'... Receiving objects: 100% (142/142), done. ← full history, on your disk
Like photocopying a colleague's entire completed lab record — every page, every revision — to build on it yourself.
The Daily Loop

status → add → commit

git status
Shows what changed: modified, new (untracked), or staged files. Your "where am I" command — run it constantly.
git add <file>
Moves a change into the staging area. Use git add . to stage everything changed.
git commit -m "message"
Permanently saves the staged snapshot to history with a description.
Edit a file
→
git status
→
git add
→
git commit
$ git status On branch main modified: syllabus.md $ git add syllabus.md $ git commit -m "Update Unit 4 references for AY 2026-27" [main 9c3a21f] Update Unit 4 references for AY 2026-27 1 file changed, 6 insertions(+), 2 deletions(-)
Real world: Updating a lab manual — you edit Experiment 5 (working dir), decide it's ready (git add), then save it as an official revision with a note: "Updated Experiment 5 — added safety precautions" (git commit -m). This loop repeats all day, every day.
Inspecting History

Looking Back — log, diff, show

git log
Lists all commits — author, date, message, unique ID (hash).
git log --oneline --graph
A compact, visual view of history — great for seeing branches.
git diff
Shows exact line-by-line changes not yet staged.
git diff --staged
Shows changes that ARE staged, about to be committed.
git show <commit>
Shows full details and changes of one specific commit.
git blame <file>
Shows who last changed each line, and in which commit.
$ git log --oneline --graph * 9c3a21f (HEAD -> main) Update Unit 4 references * 4a1f9c2 Add safety note to Experiment 5 * 7d02e88 Initial syllabus upload ← oldest commit
Real world: A student's marks look wrong in the results sheet. git blame shows exactly who edited that row, and git show reveals exactly what changed and why — a complete audit trail.

Branching & Merging

Git's superpower: working on multiple ideas in parallel without them interfering with each other.

Concept

Why Branches Matter

What is a branch?

A branch is an independent, movable line of development — a parallel copy of the project where you can experiment freely.

The main (or master) branch is the stable, "always working" version. Create a new branch to build a feature or try an idea, then merge it back once ready.

Real world example

You're revising a syllabus. You don't want to risk the version currently in use by students.

  • main branch → the syllabus currently approved & published
  • syllabus-revision-2026 branch → your draft with proposed changes
  • Once the Board of Studies approves it → merge into main
Commands

Working With Branches

git branch
List all local branches; current one is marked with *.
git switch <name>
Switch to an existing branch (modern command).
git switch -c <name>
Create AND switch to a new branch in one step.
git merge <name>
Merge the named branch's changes into your current branch.
git branch -d <name>
Delete a branch once it's been merged in safely.
git branch -a
List all branches, including remote-tracking ones.
$ git switch -c syllabus-revision-2026 Switched to a new branch 'syllabus-revision-2026' $ git branch main * syllabus-revision-2026 ← * marks your current branch
When Things Collide

Merge Conflicts — What & How

What causes it?

Two branches changed the same lines of the same file differently. Git can't decide which version is "right" — it asks you.

Real world: You and a co-faculty member both edited Question 5 of the same question paper in your own branches — one rewords it, the other changes the marks. Git flags this exact line for a human decision.

How to resolve

  1. Open the file, read both versions
  2. Manually edit to keep the correct/combined content
  3. Remove the conflict markers
  4. git add <file> to mark it resolved
  5. git commit to complete the merge
questionpaper.md <<<<<<< HEAD (main) Q5. Explain normalization in DBMS. (5 marks) ======= Q5. Explain database normalization with an example. (8 marks) >>>>>>> co-faculty-edits

Part 3

Working With Remotes

Your local repository is powerful alone — but the magic happens when it talks to a remote copy, like one on GitHub.

Local ⇄ Remote

Syncing With a Remote Repository

$ git push origin main Enumerating objects: 5, done. main -> main ← teammates can now git pull this
git remote add origin <url>
Link your local repo to a remote one, named "origin".
git remote -v
View the remotes currently configured.
git push / git pull origin main
Upload local commits, or download & merge remote changes.
git fetch
Download remote changes WITHOUT merging — inspect first.
"Oops" Commands

The Undo Toolkit

SituationCommandEffect
Undo edits in a file, not yet stagedgit restore <file>Reverts file to last committed version
Accidentally staged a filegit restore --staged <file>Unstages it, keeps your edits
Undo the last commit, keep changesgit reset --soft HEAD~1Commit removed, changes stay staged
Completely discard last commit & changesgit reset --hard HEAD~1⚠️ Destructive — changes are gone
Undo a commit that's already shared/pushedgit revert <commit>Safely adds a NEW commit that undoes it
Need to switch tasks but not ready to commitgit stashTemporarily shelves your unfinished changes
Bring shelved changes backgit stash popRestores the most recent stash
Golden rule: Once a commit is pushed and others may have it, prefer git revert over git reset — rewriting shared history causes chaos for collaborators.
Best Practices

.gitignore & Good Commit Messages

📄 .gitignore

A file listing things Git should never track — temp files, passwords, build outputs, large datasets.

*.log
node_modules/
.env
__pycache__/
Like telling a photocopier "don't copy the scrap pages and rough notes" — only the real document gets archived.

✍️ Commit Messages

"fixed stuff", "update", "asdsa", "final final"

Good: "Fix mark calculation bug in CO2 evaluation sheet"

  • Short, present-tense summary line
  • Explain why, not just what
  • One logical change per commit

Part 4

GitHub Essentials

Git gives you version control. GitHub turns it into a collaboration and project-management platform.

Getting Connected

From Local Folder to GitHub Repository

Option A: Start on GitHub

  1. Click "New repository" on GitHub
  2. Name it, add a README, choose visibility
  3. git clone <url> on your machine
  4. Add files, commit, push

Option B: Start Locally

  1. git init in your project folder
  2. Create the empty repo on GitHub (no README)
  3. git remote add origin <url>
  4. git push -u origin main
Tip: -u (or --set-upstream) links your local branch to the remote one, so future git push / git pull need no extra arguments.
Collaboration Model

Fork vs. Clone, and the Pull Request

Fork vs. Clone

Clone = download a copy to your machine (you may not have permission to push back).

Fork = your own copy of someone else's repo on GitHub itself — you fully own it and can propose changes back.

Pull Request (PR)

A formal request: "Please review and merge my branch's changes into your branch/repo."

This is GitHub's core collaboration mechanism — code review happens here, line by line, with comments.

Project Management

Issues, Labels & Project Boards

🐛

Issues

Track bugs, tasks, or ideas as individual items with discussion threads.

🏷️

Labels & Milestones

Categorize ("bug", "enhancement") and group issues into deadlines/releases.

📋

Projects (Kanban)

Drag-and-drop boards — "To Do / In Progress / Done" — built into the repo.

📥 To Do

Design database schema
Write API documentation

🔨 In Progress

Build login page — @student2

✅ Done

Set up repository
Add project README
Real world: Coordinating a final-year project team of 4 students? Create an Issue for each module ("Build login page", "Design database schema"), assign owners, and watch the Project board move from To Do to Done — exactly how industry teams manage software work.
Automation & Polish

GitHub Actions, Pages & Wiki

⚙️

GitHub Actions

Automate tasks on every push — run tests, check code style, even auto-grade assignments.

Like an automatic invigilator that re-checks every submission instantly.
🌍

GitHub Pages

Host a free website directly from a repository — perfect for a portfolio or project demo page.

📖

Wiki & README

Built-in documentation space for a project — explain setup, usage and design decisions.

Mentoring Students

How to guide students through Git/GitHub for assignments, projects, and building an interview-ready portfolio.

A Workflow to Teach

Standard Workflow for Assignments & Projects

Fork class repo
→
Clone locally
→
git switch -c assignment-1
→
Code + commit often
→
git push
→
Open PR for submission
→
Faculty reviews via comments
Why this is powerful for grading: The Pull Request becomes the submission. You can comment directly on specific lines of code, see exact commit timestamps (useful for late-submission policies), and students build real industry habits from day one.
Campus Placements

Building a Campus-Interview-Ready Portfolio

What Recruiters Look At

  • 📌 Pinned repositories — their 6 best projects, curated
  • 📝 Quality README per project — what it does, tech used, how to run it
  • 🧩 A profile README (special username/username repo) — a personal landing page

Guidance to Give Students

  • Commit regularly (not one giant commit the night before)
  • Write descriptive commit messages — recruiters do read history
  • Showcase 3-4 polished projects rather than 30 empty ones

🟩 The Contribution Graph Recruiters Check

Each square = one day's activity. Consistent small commits (not one burst before a deadline) is what this graph should show — and what recruiters notice in seconds.

Coaching Checklist

Do's & Don'ts to Teach Students

✅ Do

  • Commit small, logical chunks of work
  • Pull before you start working each session
  • Use branches for every new feature/assignment part
  • Write a README for every project
  • Resolve conflicts by reading both sides carefully

🚫 Don't

  • Commit passwords, API keys, or .env files
  • Use git push --force on a shared/team branch
  • Make one giant commit with "final project" as the message
  • Work for weeks without committing anything
  • Panic and delete the repo when a conflict appears
